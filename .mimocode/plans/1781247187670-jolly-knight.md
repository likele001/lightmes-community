# 计时工资功能实现方案

## 需求
- 计时工资计算：默认按打卡时长，未打卡时按班次时长兜底
- 时薪设置：在员工档案中设置每个员工的时薪
- 混合计薪：同一个月内，员工可以同时有计件和计时工资，月底合并

## 现有代码分析

### 工资计算流程
1. 员工报工 → 审核通过(qc_approve) → 自动触发 `calc_and_create_salary()`
2. 查 `ProcessPrice(sku_id, process_id)` 获取计件单价
3. 创建 `SalaryItem(amount = unit_price × good_qty)`
4. 月末 `calc_salary_slip_amounts()` 汇总所有 SalaryItem → SalarySlip

### 关键文件
| 文件 | 作用 |
|------|------|
| `models/salary.py` | SalaryItem 模型（report_id, sku_id, process_id, unit_price, good_qty, amount） |
| `models/salary_slip.py` | SalarySlip 模型（item_amount, bonus_amount, deduction_amount, net_amount） |
| `crud/report.py:110` | `calc_and_create_salary()` — 批量报工→工资 |
| `crud/report_unit.py:202` | `calc_and_create_salary_for_unit()` — 件次报工→工资 |
| `crud/salary_slip.py:18` | `calc_salary_slip_amounts()` — SUM(SalaryItem.amount) |
| `crud/salary_slip.py:60` | `ensure_salary_slip()` — 确保工资条存在并更新金额 |
| `crud/salary_ledger.py` | `list_salary_ledger()` — 工资明细台账（合并两种报工） |
| `models/attendance.py` | AttendanceRecord（check_in_at, check_out_at，无工时字段） |
| `models/shift.py` | Shift（start_time, end_time, rest_minutes） |
| `models/shift.py` | ShiftSchedule（user_id, shift_id, work_date） |

### 设计要点
- `calc_salary_slip_amounts` 只是 SUM(amount)，不需要改——加了 `item_type` 后自然汇总
- `SalaryItem` 的 `sku_id`/`process_id` 当前 `nullable=False`，需改成 `nullable=True`
- 唯一约束 `(tenant_id, report_id)` 和 `(tenant_id, report_unit_id)` 中 NULL 不参与，多条计时记录不会冲突
- 计时工资按日生成 SalaryItem，与计件按报工生成的模式一致

## 一、数据库改动

| hourly_rate | Numeric(12,4) | nullable | 时薪（元/小时） |

## 四、核心流程

```
月末触发
  ├─ 遍历 salary_type ∈ {hourly, mixed} 的员工
  │   ├─ generate_time_salary_items()
  │   │   ├─ 遍历当月每一天
  │   │   │   ├─ 查考勤 → 有打卡？→ calc_daily_hours(check_in, check_out)
  │   │   │   │           └─ 无 → 查排班 → calc_daily_hours(shift)
  │   │   │   └─ 生成 SalaryItem(item_type="hourly")
  │   │   └─ flush
  │   └─ ensure_salary_slip()
  │       ├─ 汇总 piece_amount + hourly_amount
  │       └─ net = item + bonus - deduction
  └─ 员工查看工资条 → 分区展示
```

## 五、优先级建议（按业务价值排序）

### P0 - 核心缺失（建议优先补齐）
1. **委外管理** — 许多工厂依赖外协加工，完全缺失
2. **审批流引擎** — 当前仅报工有硬编码审核，需要通用审批流
3. **库存成本计算** — 无法做财务核算

### P1 - 重要增强
4. **BOM版本管理+成本计算** — 版本追溯和成本核算是MES核心需求
5. **班次管理** — 排班是计件工资和产能计算的基础
6. **合同管理** — 合规和财务需求
7. **销售报价单** — 销售流程闭环

### P2 - 功能完善
8. **MRP增强**（净需求/多级BOM拆解/安全库存）
9. **APS增强**（设备级约束）
10. **工装管理**
11. **库存调拨/盘点完善**
12. **OA公告**
13. **销售评审单/核销单**
14. **质检深度增强**（首件检验/过程巡检/检验标准AQL）

### P3 - 优化提升
15. **系统监控**（服务状态/CPU/内存/磁盘）
16. **班组/工位/产线管理**
17. **设备统计增强**（运行时长/稼动率/故障率）
18. **移动端功能增强**
