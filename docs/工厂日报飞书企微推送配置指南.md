## 一、功能说明

### 1.1 什么是工厂日报

**工厂日报**（系统内称 **每日生产简报**，事件码 `brief.daily`）由 Celery 定时任务自动生成，汇总当日关键经营与生产指标，推送给厂长、管理层及相关群组。

**规则版示例**（不依赖 AI）：

```text
【2026-06-07 生产简报】
今日合格产量：120
待审核报工：5 条
逾期订单：2 单
已确认订单：8 单
待开工任务：3 个
```

**LLM 版**：在自动化设置中选择 `mode: llm` 后，由 AI 网关根据同一天的数据生成不超过约 200 字的早报（需配置 AI 模型；失败时自动回退规则版）。

### 1.2 数据来源

| 指标 | 来源 |
|------|------|
| 今日合格/不良产量 | 当日 `qc_approved` 报工汇总 |
| 待审核报工 | 批量报工 + 件次报工中 `submitted` / `leader_approved` 数量 |
| 逾期订单 | 状态为已确认/生产中且交期早于今天的订单 |
| 已确认订单数 | 仪表盘订单统计 |
| 待开工任务 | 状态为 `pending` 的任务数 |

实现代码：`backend/app/services/daily_brief.py`。

### 1.3 推送通道一览

日报触发后，系统会走 **三条并行通道**（能开几条取决于你的配置）：

| 通道 | 接收对象 | 必要条件 |
|------|----------|----------|
| **站内通知** | 拥有 `ai.use` 权限的账号 | 开启 `briefing.daily_enabled` |
| **飞书群** | `管理群` + `全厂群` | 飞书总开关开 + 群 `chat_id` 已填 + 机器人在群内 |
| **企微群** | `管理群` + `全厂群` | 企微总开关开 + 群 Webhook 已填 |
| **钉钉群** | `管理群` + `全厂群` | 钉钉总开关开 + 群 Webhook 已填 |

> **重要**：日报事件在代码中归类为 **「仅群推送」**（`GROUP_ONLY_EVENTS`），**不会**像报工待审那样按「老板 / 班组长」规则逐个推送到个人飞书或企微会话。  
> 若希望管理层在 IM 里收到日报，请将其加入 **管理群** 或 **全厂群**；若只希望部分人看，可只配置其中一个群并把对应人员拉入。

## 二、推送时间与开关

### 2.1 定时任务

| 项目 | 值 |
|------|-----|
| Celery 任务名 | `ai.daily_brief.send` |
| Beat 调度 | 每天 **20:00**（`Asia/Shanghai`） |
| 定义位置 | `backend/app/celery_app.py` → `ai-daily-brief-send` |

### 2.2 租户级开关

路径：**系统管理 → 生产自动化设置**（`/system/automation-settings`）  
权限：`setting.manage`

| 配置项 | 字段 | 说明 |
|--------|------|------|
| 生产自动化总开关 | `enabled` | 必须为 `true`，简报相关能力才生效 |
| 每日简报定时推送 | `briefing.daily_enabled` | 关闭则该租户不发送日报 |
| 生成方式 | `briefing.mode` | `rule`（规则）或 `llm`（AI 撰写） |
| 推送小时 | `briefing.daily_hour` | **界面可填，但当前后端固定 20:00 发送，此字段暂未生效** |

小程序管理端：**智能中心 → 生产自动化** 中有相同开关。

### 2.3 推荐配置（标准厂）

```json
{
  "enabled": true,
  "briefing": {
    "daily_enabled": true,
    "daily_hour": 8,
    "mode": "rule"
  }
}
```

保存后无需重启后端；**Celery Beat 必须已在运行**（见第五节）。

## 三、配置步骤（完整流程）

按下列顺序操作，缺一不可。

### 步骤 1：开启日报

1. 登录 PC 管理端（需 `setting.manage`）
2. 进入 **系统管理 → 生产自动化设置**
3. 打开 **总开关**
4. 打开 **每日简报 → 定时推送**
5. 选择 **规则汇总**（推荐）或 **LLM 生成**
6. 点击保存

**验证（即时预览，不触发推送）**：

- 首页仪表盘「今日简报」卡片
- 或调用 `GET /api/admin/ai/brief`（需 `ai.use`）

### 步骤 2：启动 Celery Worker + Beat

日报与飞书/企微异步发送均依赖 Celery。

```bash
cd /www/wwwroot/lightmes/backend   # 按你实际路径

# 检查是否在跑
ps aux | grep 'celery -A app.celery_app'

# 未运行则按宝塔部署文档第八节启动；更新代码后务必重启
# 示例（以你服务器 scripts 为准）：
# bash scripts/stop-celery.sh && bash scripts/start-celery.sh
```

**确认 Beat 已注册日报任务**：

```bash
grep daily-brief /tmp/lightmes-celery/beat.log
# 期望在每天 20:00 附近出现：
# Scheduler: Sending due task ai-daily-brief-send (ai.daily_brief.send)
```

**确认飞书/企微 flush 任务**（静默时段消息靠此发出）：

```bash
grep -E 'feishu-flush|wecom-flush' /tmp/lightmes-celery/beat.log
```

详见 [宝塔部署.md](./宝塔部署.md) 第八节。

### 步骤 3：配置群组（飞书 + 企微，推荐用消息中心）

路径：**系统管理 → 消息中心**（`/system/message-center`）  
权限：`setting.manage`

系统预置 **3 个逻辑群**（code 固定，推送规则按 code 匹配）：

| code | 名称 | 日报是否使用 | 典型用途 |
|------|------|--------------|----------|
| `production` | 生产群 | 否 | 报工待审等业务通知 |
| `management` | 管理群 | **是** | 日报、预警、自动化失败等 |
| `factory` | 全厂群 | **是** | 日报、严重预警广播 |

在 **群组配置** Tab 为 `management`、`factory` 填写：

| 通道 | 填写内容 | 获取方式 |
|------|----------|----------|
| **飞书** | `chat_id`（`oc_xxx`） | 飞书推送页「拉取机器人所在群」 |
| **企微** | 群机器人 Webhook 完整 URL | 企微群内添加机器人后复制 |

每个群可 **同时** 开飞书和企微，填完后点击页面底部 **保存** 按钮（会同步写入飞书/企微配置）。

> **勿把管理后台网址填进飞书 chat_id**。chat_id 必须是 `oc_` 开头的飞书群 ID，不是 `https://admin...` 这类 URL。

> 也可分别在 **飞书消息推送**、**企微消息推送** 独立页维护群组；与消息中心保存效果相同。

### 步骤 4：开启飞书推送（若用飞书）

完整对接见 **[飞书消息推送部署指南.md](./飞书消息推送部署指南.md)**。日报相关最小清单：

1. 飞书开放平台创建企业自建应用，开通消息、群组相关权限
2. 将应用机器人拉入 **管理群**、**全厂群**
3. Admin → **飞书消息推送**：
   - 开启 **启用飞书推送**
   - 填写 App ID / App Secret
   - **测试连接** 成功
4. 消息中心填入两群的飞书 `chat_id`
5. 更新飞书相关代码或迁移后 **重启 Celery**

推送规则 Tab 中 **每日生产简报 `brief.daily`** 建议保持 **启用**（用于展示与部分配置同步；实际群路由见本文 1.3 节）。

### 步骤 5：开启企微推送（若用企微）

路径：**系统管理 → 企微消息推送**（`/system/wecom-notify`）

1. 填写 CorpID、AgentId、Corp Secret 等
2. 开启 **启用企微推送**
3. **测试连接** 成功
4. 在目标企微群添加 **群机器人**，复制 Webhook
5. 在消息中心对应群的 **企微通道** 粘贴 Webhook 并启用
6. 重启 Celery（若刚上线企微功能）

### 步骤 6：指定「相关人员」如何收到

根据你的组织方式选择：

| 目标 | 推荐做法 |
|------|----------|
| 厂长 / 管理层看 **站内** 日报 | 给用户角色分配 **`ai.use`** 权限 |
| 飞书 **群里** 收日报 | 将人员拉入已配置 `chat_id` 的 **管理群 / 全厂群** |
| 企微 **群里** 收日报 | 将人员拉入已配置 Webhook 的 **管理群 / 全厂群** |
| 飞书 / 企微 **个人会话** 收日报 | **当前版本不支持** 自动推个人；请进群或看站内通知 |
| 仅管理层看、不想全厂扩散 | 只配置 **管理群**，关闭或留空 **全厂群** 的通道 |

**与报工待审个人推送的区别**：

- 报工待审（`report.submitted`）：走规则目标（班组长等），且员工需 **绑定飞书/企微** 才能收个人消息
- 工厂日报（`brief.daily`）：只推 **管理群 + 全厂群**，不走个人 open_id / userid

员工绑定飞书/企微：H5 或小程序 **个人中心 → 飞书/企微绑定**（日报不依赖此项，但报工审核个人通知依赖）。

## 四、消息中心各 Tab 说明

| Tab | 与日报的关系 |
|-----|----------------|
| 通道总览 | 查看飞书/企微是否已配置、今日推送量 |
| 群组配置 | **必配**：`management`、`factory` 的 chat_id / Webhook |
| 推送规则 | 查看 `brief.daily` 等事件；日报群路由以代码 `EVENT_GROUP_CODES` 为准 |
| 人员绑定 | 主要用于报工等待审等 **个人** 推送，日报不强制 |
| 推送日志 | 筛选 `event_code = brief.daily` 查成功/失败 |
| 告警接收人 | 用于 **推送连续失败** 时通知超管，**不是** 日报订阅名单 |

## 五、技术细节（给实施 / 运维）

### 5.1 发送链路

```text
Celery Beat 20:00
  → ai.daily_brief.send
    → send_daily_briefs_for_all_tenants (遍历租户)
      → send_daily_brief_for_tenant (检查 daily_enabled)
        ├─ notify_users_with_permission(ai.use)  → 站内通知
        └─ dispatch(event_code="brief.daily")   → 飞书群 + 企微群
             → feishu.send_message / wecom.send_message (Celery 异步)
```

### 5.2 群路由代码

`backend/app/services/notify_channels.py`：

```python
EVENT_GROUP_CODES = {
    "brief.daily": ["management", "factory"],
    ...
}
```

即：只要飞书/企微通道开启且群组 enabled，就会向这两个 code 对应的群各发一条。

### 5.3 静默时段

若在飞书/企微设置中开启 **静默时段**（如 22:00–07:00），非紧急消息可能标记为 `deferred`，由 Beat 任务 `feishu.flush_deferred` / `wecom.flush_deferred`（每 5 分钟）在时段结束后发出。  
日报固定在 **20:00**，通常不在默认静默窗口内，会立即进入 `pending` 发送。

### 5.4 API 速查

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/admin/automation/settings` | 读取自动化含 briefing |
| PUT | `/admin/automation/settings` | 保存 daily_enabled / mode |
| GET | `/admin/ai/brief` | 实时生成简报（不推送） |
| GET | `/admin/ai/brief/latest` | 最近一条已发送简报（站内） |
| GET | `/admin/system/message-center/overview` | 通道总览 |
| GET | `/admin/system/feishu` | 飞书配置 |
| GET | `/admin/system/wecom` | 企微配置 |

上线前逐项打勾：

- [ ] 生产自动化 **总开关** 与 **每日简报定时推送** 已开启
- [ ] Celery **Worker + Beat** 运行中，`beat.log` 可见 `ai-daily-brief-send`
- [ ] 消息中心 **管理群、全厂群** 已填飞书 `chat_id` 和/或企微 Webhook

- [ ] 飞书/企微 **总开关** 已开，测试连接成功
- [ ] 至少一名管理账号拥有 **`ai.use`**（收站内简报）
- [ ] 等到 **20:00** 后检查推送日志 `brief.daily` 为 `success`
- [ ] 或临时在服务器手动触发（仅测试环境）：

```bash
PYTHONPATH=. python -c "
from app.core.db import SessionLocal

"
```

## 七、常见问题

| 现象 | 可能原因 | 处理 |
|------|----------|------|
| 完全没收到 | `daily_enabled` 关闭 | 生产自动化设置中打开 |
| 完全没收到 | Celery Beat 未运行 | 启动并检查 `beat.log` |
| 完全没收到 | 未到 20:00 | 等待定时或用手动触发测试 |
| 只有站内、无飞书/企微 | 通道总开关关闭 | 飞书/企微页开启并测试连接 |
| 飞书/企微无消息 | 群 `chat_id` / Webhook 为空 | 消息中心补全 management、factory |
| 飞书群无消息 | 机器人未进群 | 飞书群设置中邀请应用机器人 |
| 企微群无消息 | Webhook 失效或 URL 错误 | 重新创建群机器人并更新 URL |
| 个人飞书/企微没收到 | 日报不推个人 | 将人拉入管理群/全厂群，或看站内 |
| 改了推送小时无效 | `daily_hour` 未接入 Beat | 当前固定 20:00；需改代码或联系开发 |
| LLM 简报失败 | AI 网关未配 | 改用 `mode: rule` 或配置 AI 模型 |
| 日志 `deferred` 一直不发 | flush 任务未跑 | 重启 Celery，查 `feishu-flush-deferred` |
| 消息中心保存群组 **405 Method Not Allowed** | 线上后端未更新到含 `POST/PUT .../groups` 的版本 | 拉取最新代码并 **重启 FastAPI**；见下文 |

推送失败可在 **消息中心 → 推送日志** 或飞书/企微独立日志页查看 `error_msg`，对 `failed` 记录可点 **重试**。

### 消息中心保存报 405 时

浏览器请求 `POST /api/admin/system/message-center/groups` 若返回 `{"detail":"Method Not Allowed"}`，说明 **当前运行的后端仍是旧版**（只有 `GET /groups`，没有保存接口）。

**处理：**

```bash
cd /www/wwwroot/lightmes          # 服务器项目目录
git pull                          # 或你的发布方式
cd backend && alembic upgrade head
# 宝塔：Python 项目管理 → lightmes → 重启
# 或手动重启 uvicorn/gunicorn
```

重启后应用日志或 OpenAPI 中应能看到 `POST /admin/system/message-center/groups`。

**临时替代**：在 **飞书消息推送** / **企微消息推送** 独立页的「群组管理」里填写并保存，效果相同。

## 八、与其他通知的对比

| 事件 | 事件码 | 推送方式 |
|------|--------|----------|
| 派工通知 | `dispatch.assigned` | 被派工员工 **个人**（需绑定） |
| 报工待审 | `report.submitted` | 班组长/车间负责人 **个人** + 规则 |
| AI/业务预警 | `alert` | 按级别升级，含 **管理群、全厂群** |
| **每日生产简报** | **`brief.daily`** | **仅管理群 + 全厂群** + 站内 `ai.use` |
| 自动化失败 | `plan.automation_failed` | 计划权限 + **管理群** |
