# LightMes 服务器迁移 / 重新部署教程

> 适用场景：**把 LightMes 从旧服务器完整迁到新服务器**（重装系统、换机器、机房迁移等），以及在新机器上**从零部署**。
>
> 本文是"**必做项**"清单——按顺序走完就能跑通。任何一步漏了都可能导致：**飞书/钉钉推送静默失效、Celery 任务积压、和同机其他项目（bizcloud / dify）相互串台**。
>
> 配套文档：
> - [宝塔部署.md](./宝塔部署.md) — 基础环境、Nginx、MySQL、Redis
> - [宝塔部署常见错误排查.md](./宝塔部署常见错误排查.md) — 503/缺包/端口/Redis
> - [飞书消息推送部署指南.md](./飞书消息推送部署指南.md) — 飞书 OAuth / 群 / 事件订阅
> - [钉钉消息推送部署指南.md](./钉钉消息推送部署指南.md) — 钉钉群机器人 / 工作通知

## 0. 迁移前的 3 个关键判断

迁之前**先回答**这三个问题（漏一个后面会出大问题）：

1. **新机器上还有别的项目用 Redis 吗？**（常见：dify、bizcloud）
   - 有 → 必须**分 Redis 库**（见 §5）
2. **新机器上还有别的项目用 Celery 吗？**
   - 有 → Celery 任务名不能撞（见 §5.2）
3. **飞书/钉钉/企微 的 AppID/AppSecret 跨服务器会变吗？**
   - 不会变 → 沿用旧 `.env` / 租户设置
   - 会变（重新创建应用）→ 旧 `open_id` 全部失效，必须重做 OAuth 绑定

## 1. 备份旧服务器的关键数据

迁移要搬走的就 4 类东西：

```bash
# 1.1 整个项目目录（含 .env、uploads、storage）
tar -czf /www/backup/lightmes_src_$(date +%Y%m%d).tar.gz \
  /www/wwwroot/lightmes \
  --exclude='/www/wwwroot/lightmes/frontend-*/node_modules' \
  --exclude='/www/wwwroot/lightmes/frontend-*/dist' \
  --exclude='/www/wwwroot/lightmes/.git'

# 1.2 MySQL 整库
mysqldump -uroot -p --single-transaction --routines --triggers lightmes \
  | gzip > /www/backup/lightmes_db_$(date +%Y%m%d).sql.gz

# 1.3 文件存储（报工图片/视频/导出 Excel）
tar -czf /www/backup/lightmes_storage_$(date +%Y%m%d).tar.gz \
  /www/wwwroot/lightmes/backend/data/storage

# 1.4 环境变量（最关键的就是 .env 和 feishu/dingtalk 凭据）
cp /www/wwwroot/lightmes/backend/.env /www/backup/lightmes.env.bak
```

把上面 4 个文件 scp 到新机器：

```bash
scp /www/backup/lightmes_*.{tar.gz,sql.gz,env.bak} root@新服务器IP:/www/backup/
```

## 2. 新机器基础环境（宝塔裸机，**不要用 Docker**）

> CLAUDE.md 明确要求：**生产部署默认宝塔裸机，禁止 Docker/K8s 作为唯一生产方案**。本文档按裸机写。Docker 仅用于本地开发（见 [Docker部署指南.md](./Docker部署指南.md)）。

宝塔面板装好后，按顺序装：

| 软件 | 版本要求 | 用途 | 必装 |
|---|---|---|---|
| Nginx | 1.20+ | 反代 + 静态 | ✅ |
| MySQL | **5.7+** | 业务库 | ✅ |
| Redis | 6.x / 7.x | Celery broker + 验证码 + 缓存 | ✅ |
| Python 项目管理器 | 任意 | 建 venv | ✅ |
| 宝塔 supervisor | 任意 | 守护 Celery（可选，cron 也行） | ⚠️ |

**MySQL 5.7** 严重要求：代码里用了大量 5.7 兼容语法，**禁止装 MySQL 8.0**（字符集、索引、JSON 字段行为有差异，迁过去会报奇怪的错）。

**Redis** 装完启动后必须确认能 ping 通：

```bash
redis-cli ping   # 应返回 PONG
```

```bash

#     - 项目路径: /www/wwwroot/lightmes/backend
#     - Python 版本: 3.10 或 3.11（推荐 3.10）
#     - 框架: 其他
#     - 启动方式: gunicorn / uvicorn 都行
#     - 端口: 8000

# 3.2 激活 venv 后装依赖
source /www/server/pyporject_evn/lightmes/bin/activate
cd /www/wwwroot/lightmes/backend

# 3.3 关键包单独确认（漏一个会出怪问题）
pip install celery==5.3.* redis pymysql sqlalchemy pydantic pydantic-settings \
            fastapi uvicorn python-multipart httpx openai pillow
```

> ⚠️ 一定看清：`celery 5.3.x` 和 `celery 5.5+` 默认队列名行为不一样（5.5 之前默认队列是 `celery`，5.5+ 是 task 名）。混装环境会导致**任务入队后无人消费**。

## 5. ⚠️ 必做：Redis 分库（**不踩会出大事**）

> **这次出问题的根源就在这一节**。LightMes 和同机的 bizcloud 项目同时使用 `app.celery_app` 这个 app 名 + 同一个 Redis db 0，导致：
> - lightmes 的飞书任务被 bizcloud 的 worker 当成 `unregistered task` 丢弃
> - bizcloud 的任务也压到 lightmes 队列里
> - 任务量一上去两边日志全是红，刷不到根因

### 5.1 默认规划（推荐）

| 项目 | Redis db | 说明 |
|---|---|---|
| lightmes（应用 + Celery broker + result） | **2** | 干净独立 |
| bizcloud | 0 | 沿用 docker compose 默认 |
| dify / 其他 | 1 | 沿用 dify 默认 |
| 备用 | 3 | — |

### 5.2 改 lightmes 的 `.env`

```bash
vi /www/wwwroot/lightmes/backend/.env
```

把以下三项全部改成 **db 2**（或者任何不和同机项目冲突的库）：

```ini
REDIS_URL=redis://127.0.0.1:6379/2
CELERY_BROKER_URL=redis://127.0.0.1:6379/2
CELERY_RESULT_BACKEND=redis://127.0.0.1:6379/2
```

> **注意**：`REDIS_URL` 是项目里所有 Redis 用途（验证码、缓存、限流）共用的，必须和 Celery 走同一个 db。否则会出现"应用 Redis 走 db 0 撞 bizcloud"的同类问题。

### 5.3 验证分库生效

```bash
redis-cli -n 0 DBSIZE   # 应该是 4~5（bizcloud）
redis-cli -n 1 DBSIZE   # dify 的，可能是 1w+
redis-cli -n 2 DBSIZE   # 启动后会是 3（celery + pidbox + celeryev）
```

### 5.4 防止同事故再发

如果你**未来还可能加新项目**到这台机器，把 `REDIS_URL` 的 db 号写进 `docs/` 里的"部署清单"（`docs/部署说明.md`），所有项目**必须**用各自独立的 db。

## 7. 启动 Celery（**关键三步**）

### 7.1 用项目自带脚本启动

```bash
bash /www/wwwroot/lightmes/scripts/start-celery.sh
```

**不要**自己拼 `celery -A app.celery_app worker ...` 命令行。脚本会：
- 检查 Redis 可达
- 启动 worker + beat
- 写 PID 文件到 `/tmp/lightmes-celery/`

### 7.2 验证 worker 真的连上了 db 2

```bash
tail -20 /tmp/lightmes-celery/worker.log | grep -E 'Connected to redis|celery@.* ready'
```

必须看到 `Connected to redis://127.0.0.1:6379/2`（**不是 db 0**）。

### 7.3 验证关键任务都注册了

```bash
grep -E '^\s+\.' /tmp/lightmes-celery/worker.log | head -20
```

应能看到 `feishu.send_message`、`feishu.flush_deferred`、`dingtalk.send_message`、`wecom.send_message` 等 10+ 个任务。

### 7.4 验证 Beat 调度

```bash
tail -30 /tmp/lightmes-celery/beat.log
```

应能看到 `feishu-flush-deferred`、`wecom-flush-deferred`、`dingtalk-flush-deferred`、`ai-daily-brief-send` 每 5 分钟 / 每天 20:00 调度。

## 8. 开机自启（**必做！不设重启就丢**）

LightMes **没有 systemd unit / supervisor 默认配置**，必须手动加。

### 方案 A：crontab `@reboot`（最省事）

```bash
crontab -e
# 加两行：
@reboot /bin/bash -lc 'sleep 8 && /www/wwwroot/lightmes/scripts/start-celery.sh >> /tmp/lightmes-celery/boot.log 2>&1'
*/5 * * * * /bin/bash -lc '/www/wwwroot/lightmes/scripts/start-celery.sh >> /tmp/lightmes-celery/boot.log 2>&1'
```

- `@reboot` 开机后等 8 秒（让 Redis 起来）再拉 Celery
- `*/5` 是兜底：进程挂了 5 分钟自检拉起

> 配套的 `scripts/start-celery.sh` 已经做了**幂等检查**（lightmes venv 的 worker + beat 都在就跳过），不会双跑。

### 方案 B：宝塔 supervisor（更稳，推荐生产用）

宝塔面板 → 软件商店 → 宝塔 supervisor → 添加守护进程：

| 名称 | 启动命令 | 运行用户 |
|---|---|---|
| lightmes-celery | `bash /www/wwwroot/lightmes/scripts/start-celery.sh` | root |

> **注意**：不要把 worker 和 beat 拆成两个进程，脚本里一起起的。supervisor 看到的"主进程"是 nohup 出去的 worker 的父 shell，**可能识别不到 worker 子进程是否存活**。如果遇到这问题，改成 `Type=oneshot` + `RemainAfterExit=yes` 即可。

### 验证

```bash
# 重启机器后
reboot
# 登录后 1 分钟内
ps aux | grep 'celery -A app.celery_app worker' | grep lightmes
# 应能看到 1+ worker + 1 beat
```

## 9. Nginx 反代

宝塔站点 → 添加站点 → 域名填 `admin.example.com` → 配置文件加：

```nginx
server {
    listen 80;
    server_name admin.example.com;
    client_max_body_size 200M;     # 报工视频上传必须 ≥ 100M

    # PC 前端静态资源（vite build 产物）
    location / {
        root /www/wwwroot/lightmes/frontend-admin-pro/dist;
        try_files $uri $uri/ /index.html;
    }

    # H5 员工端（如启用）
    # location /h5/ { alias /www/wwwroot/lightmes/frontend-h5/dist/; try_files $uri $uri/ /h5/index.html; }

    # API 反代
    location /api/ {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 120s;
    }
}
```

SSL 在宝塔站点 → SSL → Let's Encrypt 一键签。

> 飞书事件订阅 URL 是 `https://你的域名/api/feishu/events`，**必须是公网 HTTPS**，本地 / 内网 IP 飞书无法访问。

## 10. 飞书 / 钉钉 / 企微 推送（跨机最易丢的环节）

> 完整步骤见 [`飞书消息推送部署指南.md`](./飞书消息推送部署指南.md) 和 [`钉钉消息推送部署指南.md`](./钉钉消息推送部署指南.md)。这里只列**跨服务器迁移的注意点**。

### 10.1 凭据在哪

凭据存在 **两个地方**，迁移时两个都要搬：

| 位置 | 内容 |
|---|---|
| `backend/.env` | `FEISHU_APP_ID`、`FEISHU_APP_SECRET`、`DINGTALK_APP_KEY` / `DINGTALK_APP_SECRET`、各种 webhook |
| 数据库 `tenant_settings` 表 `key='feishu.notify' / 'dingtalk.notify'` | 群组 chat_id / webhook / 推送规则 / 人员绑定 open_id |

`mysqldump` 已经把后者搬过来了，**只要 .env 还在就够**。

### 10.2 哪些**不会**自动恢复

| 项 | 跨服务器表现 | 处理 |
|---|---|---|
| 飞书 `open_id` | **失效**（飞书按 App 隔离，AppID 不变则不变；变了必须重绑） | Admin → 飞书推送 → 「按手机号批量匹配」一键还原 |
| 钉钉 webhook 加签 secret | 沿用 | 无需处理 |
| 企微机器人 webhook | 沿用 | 无需处理 |
| 飞书事件订阅 URL | **必须改**（域名变了） | 飞书开放平台 → 事件订阅 → 改 URL，重新走 challenge 验证 |
| OAuth 回调 URL | **必须改** | 飞书开放平台 → 安全设置 → 改重定向 URL |

### 10.3 飞书事件订阅验证

```bash
# 启动后端后，主动测一次
curl -X POST https://你的域名/api/feishu/events \
  -H "Content-Type: application/json" \
  -d '{"type":"url_verification","challenge":"test123"}'
# 应返回 {"challenge":"test123"}
```

### 10.4 验证推送链路

后端起来后，手动发一条：

```python
# 在 backend 目录
source /www/server/pyporject_evn/lightmes/bin/activate
python -c "
from app.celery_app import celery
r = celery.send_task('feishu.send_message', args=[<一个真实 log_id>])
print(r.id)
"
```

然后看 `/tmp/lightmes-celery/worker.log`，应能看到 `succeeded ... message_id: om_xxx`。

## 12. 这次踩过的坑（避坑备忘）

| # | 现象 | 根因 | 修复 |
|---|---|---|---|
| 1 | 飞书/钉钉推送全失败 | 容器里 Celery worker 命令行是 `-Q default,ai`，**漏了 `celery` 默认队列**，所有 send_task 全部进 `celery` 队列但无人消费 | 改 worker 启动参数订阅 `celery` 队列，或去掉 `-Q` |
| 2 | Celery worker 启动后日志里看不到 `feishu.send_message` 任务 | `app.celery_app.autodiscover_tasks(["app"])` 失败（import 报错） | 看 worker.log 找 traceback；多半是某个 tasks 文件 import 链断了 |
| 3 | lightmes 和 bizcloud 任务互相抢 | 两个项目用同一个 `app.celery_app` + 同一个 Redis db 0 | **分库**（见 §5） |
| 4 | 容器 Celery 进程 `kill` 完自动拉起 | `docker compose` 的 `restart: unless-stopped` | `docker stop`（不是 `kill`），或改 compose 删 `restart` |
| 5 | `ps` 看到的 Celery 进程 venv 路径和脚本不一致 | 早期手动 `nohup` 起的孤儿进程，start 脚本管不到 | 先 `pkill -9` 旧进程，再走脚本启动 |
| 6 | `apt install python3-celery` 后找不到 worker | 系统包和 venv 包混用 | 全部用 venv 路径（`/www/server/pyporject_evn/lightmes/bin/celery`） |
| 7 | 钉钉 ActionCard 一直 400002 | `build_action_card` 用了 `text` 字段，钉钉要的是 `markdown` | 改 `services/dingtalk/cards.py` 字段名 |
| 8 | 钉钉报错只看到 `ok: False` 不知道发出去啥 | `send_webhook_message` 失败时不打 payload | 改 `client.py` 失败时 `logger.warning(payload + response)` |
| 9 | 前端改了 `.vue` 页面没生效 | 浏览器缓存 + 没用 build 后的产物 | `npm run build`，强刷 Ctrl+Shift+R |
| 10 | 机器重启后 Celery 没起来 | 没有 `@reboot` cron | 加 `crontab -e`（见 §8） |
| 11 | 钉钉推送缺 `markdown` 字段但客户端测试又通过 | 钉钉服务端**宽松校验**，业务代码走的是工作通知 `asyncsend_v2` 路径（要求更严） | 走 `target_kind='webhook'` 走群机器人路径严格测 |
| 12 | `npm run build` 后 `dist/` 里还有旧 chunk 名 | vite 用 hash 命名，新 chunk 名是新的；旧的没人引用 | 直接 `rm -rf dist && npm run build` 重建 |

## 13. 一键健康检查脚本

部署完跑一遍，6 项全绿才算真成功：

```bash
cat > /tmp/lightmes_healthcheck.sh << 'EOF'
#!/usr/bin/env bash
RED='\033[0;31m'; GRN='\033[0;32m'; NC='\033[0m'
PASS=0; FAIL=0
check() {
  if eval "$2" >/dev/null 2>&1; then
    echo -e "${GRN}✓${NC} $1"; PASS=$((PASS+1))
  else
    echo -e "${RED}✗${NC} $1"; FAIL=$((FAIL+1))
  fi
}

check "Redis 可达"        "redis-cli ping"
check "MySQL lightmes 库"  "mysql -ulightmes -p123456 -e 'use lightmes' lightmes"
check "后端 API /api/health" "curl -sf http://127.0.0.1:8000/api/health"
check "Celery worker 进程" "pgrep -fa 'app.celery_app worker' | grep -q /www/server/pyporject_evn/lightmes/bin/celery"
check "Celery beat 进程"   "pgrep -fa 'app.celery_app beat' | grep -q /www/server/pyporject_evn/lightmes/bin/celery"
check "Worker 连 db 2"     "grep -q 'Connected to redis://127.0.0.1:6379/2' /tmp/lightmes-celery/worker.log"
check "Beat 调度 feishu"   "grep -q 'feishu-flush-deferred' /tmp/lightmes-celery/beat.log"
check "@reboot cron 配置"  "crontab -l | grep -q '@reboot.*start-celery.sh'"

echo "---"
echo -e "通过: ${GRN}$PASS${NC}  失败: ${RED}$FAIL${NC}"
EOF
chmod +x /tmp/lightmes_healthcheck.sh
bash /tmp/lightmes_healthcheck.sh
```

预期全绿。哪项红了就对照本文找对应章节。

## 14. 紧急回滚

如果新机器装完发现推飞书失败 / 业务异常，**5 分钟回滚到旧机器**：

```bash
# 14.1 DNS 切回旧服务器（或在宝塔站点把 SSL 停掉，让流量走旧）
# 14.2 旧服务器不动，业务继续
# 14.3 新服务器排查（按本文 §12 表格 + §13 健康检查）
```

回滚前**不要**删旧服务器的任何东西，至少保留 7 天。
